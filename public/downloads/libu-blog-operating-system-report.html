礼部接管总方案

“重开老赵的朝廷”
博客运营系统设计报告

这份报告为 礼部接管博客 提供一套完整的设计蓝图。它覆盖前端页面结构、后端 Agent 拓扑、脚本体系、Scale 模块、发布流程、运营节奏，以及每一页应该承载的内容与字段。目标不是做一个“能写点东西的后台”，而是做一套 便宜模型也能稳定运行 的内容操作系统。

唯一总控 Agent 礼部
推荐内容源 文件优先
推荐发布路径 脚本校验后发布
目标 稳定、可迁移、低成本

01 / 执行摘要

核心判断：礼部不应该登录网页后台，而应该操作一套被脚本约束的内容系统

你的目标不是让廉价模型去点击按钮，而是让廉价模型在固定边界内填充固定结构。最优设计不是“CMS 后台 + Agent 登录”，而是：

礼部

唯一对外运营总控。负责选题、定调、归档、发布、专题结构、SEO 口径与节奏管理。

脚本

唯一发布通道。负责新建骨架、校验字段、标准化 slug、构建数据、同步前台、触发部署。

站点

只负责展示，不承担“自由编辑”的职责。前端读规范化后的内容，而不是让模型临场发挥。

推荐主路线：文件优先 + 脚本发布 + 前端读取构建产物。评论、订阅等弱结构功能可以继续挂 Firebase；日志、工作流、智能体、产品这类核心展示内容应交由礼部脚本系统统一发布。
02 / 系统架构

推荐架构：礼部内容操作系统

下面这条链路是最适合 OpenClaw 的：让礼部汇总上游，再通过脚本发布到前端，而不是让每个 Agent 直接写线上站点。

上游输入

国子监研究、翰林院长文、兵部产品进度、内务府素材、OpenClaw 日常运行记录。

→

礼部内容中枢

统一选题、确定栏目、拆成标准内容类型：日志、工作流、智能体、产品、专题。

→

发布脚本

校验 schema、生成 slug、构建 JSON、同步前端数据源、构建部署、记录运营日志。

层 职责 推荐实现
前端层 负责展示首页、Agent 页、工作流页、日志页、产品页和下载页 继续使用 `ZZM` 现有 React + Vite 前端
内容层 负责保存结构化内容与版本历史 Git 仓库中的 `content/` 目录，或 Firestore 的规范化镜像
发布层 负责校验、变换、构建和部署 Node 脚本：`new / validate / publish / deploy / sync`
运营层 负责节奏、题库、专题与分发 礼部 Agent + 固定 Scale
动态层 负责评论、订阅、统计、少量实时指标 Firebase 保留，用于弱结构功能
不推荐把“日志、工作流、智能体、产品”长期交给网页表单人工维护。表单适合作为临时兜底，不适合作为廉价模型的主要操作面。主路径应该是脚本和规范化文件。
03 / Agent 拓扑

礼部总控，其他 Agent 供稿，发布权限集中

这里不是让所有 Agent 都能发布，而是让礼部吸收他们的产出后再对外成文。

礼部

唯一发布者 栏目总编

负责栏目规划、标题改写、栏目归属、首页编排、分发文案、SEO 标题与发布节奏。

翰林院

长文加工

负责把复杂讨论整理成长文、案例分析、技术说明。只供稿，不直接发布。

国子监

研究供稿

负责背景研究、竞品信息、引用材料、术语定义、知识补完。

兵部

产品战报

提供产品推进、实验结果、版本发布、上线信息、客户案例素材。

内务府

资产管理

维护图片、截图、下载附件、报告、配置文件、scale 包与封面资源。

都察院

风控校验

负责发布前的敏感性检查、误导风险、承诺风险、隐私与数据口径审查。

1

上游供稿

各 Agent 输出原始记录、研究笔记、产品结果、日志摘要。

2

礼部重写

礼部决定该内容属于哪一类页面，选择模板、栏目与受众口径，并生成标准结构。

3

脚本校验

脚本确认字段齐全、JSON 正确、slug 可用、日期和数组格式合法。

4

构建与发布

构建前台可读的数据产物，更新页面，部署上线，并在礼部运营日志中留痕。

04 / 礼部 Scale 设计

礼部需要的不是一个大而全提示词，而是一组窄而稳的 Scale 模块

建议把礼部的 Scale 拆成 5 个模块，每个模块都只负责一种工作状态。

01. `editorial-governor`

用途：决定内容类型、受众、栏目、标题、摘要和 CTA。

输入：原始记录、页面目标、内容类型。

输出：标准 frontmatter / JSON 草稿。

02. `log-stabilizer`

用途：把工作记录稳定转成“今日做了什么 / 遇到什么 / 如何解决”。

输入：日报碎片、命令输出、讨论记录。

输出：日志页面结构化内容。

03. `workflow-documenter`

用途：把一次协作抽象成工作流案例。

输入：参与 Agent、步骤、输入、输出、关键决策。

输出：长卡片与详情页数据。

04. `product-narrator`

用途：把产品做成“能卖”的案例页，不是堆功能清单。

输入：产品链接、挑战、方案、交付结果、指标。

输出：产品卡片、详情页、亮点与价格信息。

05. `release-guard`

用途：发布前最后一关，检查格式、风险、空字段、错别字、敏感说法。

输入：即将发布的结构化内容。

输出：允许发布 / 阻止发布。

推荐原则

不要让礼部一次承担“写作 + SEO + 发布 + 部署 + 复盘”。每个 Scale 只承担一个窄任务，脚本负责流程编排。

{
  "scaleName": "editorial-governor",
  "owner": "libu1",
  "purpose": "Turn raw OpenClaw outcomes into stable public-facing content.",
  "inputs": ["contentType", "rawNotes", "audience", "pageGoal"],

05 / 脚本体系

推荐脚本：让礼部只填数据，不碰危险操作

建议把礼部运维脚本独立成一组固定命令，每个命令只做一件事。

脚本 作用 输入 结果
new-log.mjs 新建日志骨架 日期、标题、摘要 生成标准日志文件
new-workflow.mjs 新建工作流案例 slug、标题、参与 Agent 生成工作流 JSON 模板
update-agent.mjs 更新智能体资料 agent id、字段 patch 更新 Agent 内容源
publish-product.mjs 发布产品案例 产品 slug、结构化数据 写入产品内容源
validate-content.mjs 校验所有内容 内容目录 发现空字段、错格式、重复 slug
build-site.mjs 构建站点 构建配置 生成前端产物
deploy-site.mjs 触发部署 git 状态 / CI 参数 推送并上线
sync-metrics.mjs 同步营收与 token 消耗 来源数据文件 更新首页和产品指标卡

06 / 页面内容设计

每一页应该承载什么内容

下面这一部分是站点的信息架构核心。礼部以后的维护动作，应该围绕这些页面定义来进行。

首页

模块 作用 来源
Hero 一句话说清 OpenClaw 是什么、礼部如何经营它、点击后去哪里 礼部固定文案
How it works 解释多 Agent 如何协作，不是抽象口号 工作流摘要 + 礼部说明
营收 / Token / 产品数 把系统能力变成可感知的经营结果 metrics/ 数据
订阅 承接感兴趣的人 订阅服务

智能体页（列表）

卡片必须展示

  • Agent 名称
  • 角色定位
  • 3 个代表技能
  • 一句话职责
  • 配置 / scale 下载入口

详情页必须展示

  • Agent 的工作边界
  • 配置摘要
  • Scale 摘要
  • 典型输入 / 输出
  • 下载链接与 GitHub 来源

工作流页（列表 + 详情）

页面 内容设计
列表页 长卡片展示问题、适用场景、参与 Agent、主要产物、点击后价值
详情页左侧 流程图或阶段卡，清楚显示谁先做、谁接手、谁定稿
详情页下方 实现说明、关键决策、输入示例、输出成果、失败边界

日志页

固定结构

今日做了什么 / 遇到哪些困难 / 如何解决 / 下一步是什么。

指标块

当天收入、token 消耗、完成任务数、重要产出。

评论区

保留给人类读者，不交给礼部处理实时互动。

产品页

列表页

产品卡片要像案例集，不像商城。重点是：做了什么、解决了什么、由谁做、结果怎样。

详情页

挑战、方案、交付物、参与 Agent、价格或营收结果、在线演示、购买入口。

后台页

后台页只作为人类兜底与调试台，不应成为礼部的主操作面。它适合人临时补字段，不适合廉价模型日常操作。礼部的主路径应该是：脚本 + 内容文件。

07 / 运营流程

礼部的日常工作流应该固定成 8 步

1

收集输入

从各 Agent 接收研究、产品进度、运行结果、失败记录、用户反馈。

2

判定内容类型

礼部判断这条输入属于日志、工作流、智能体更新、产品案例还是首页指标更新。

3

调用对应 Scale

礼部只调用当前类型需要的那个 Scale，不做大杂烩生成。

4

生成草稿文件

由脚本创建标准骨架，礼部往里填，不手写乱序字段。

5

运行校验

检查 slug、日期、数组、JSON、空字段、指标来源、敏感风险。

6

预览

本地构建站点，检查版式、段落长度、卡片摘要与页面跳转。

7

发布

脚本写入站点内容源，触发构建与部署，上线后留痕。

8

复盘

礼部记录本次发布时间、目标、表现、异常与下一次优化方向。

09 / 实施顺序

推荐分 3 期推进

第 1 期：定标准

完成内容目录、JSON schema、礼部 Scale、基础脚本框架。

目标：礼部能稳定产出而不乱格式。

第 2 期：定流程

让 `new / validate / publish / deploy / sync-metrics` 脚本跑通，建立标准发布节奏。

目标：礼部可独立运营站点。

第 3 期：定经营

接真实 token 消耗、营收数据、订阅与统计，形成“产品演示 + 公开运营日志”。

目标：让博客变成可销售的样板间。

结论：礼部负责运营，脚本负责秩序，前端负责展示，Firebase 只保留动态与兜底能力。 这样才是最适合 OpenClaw、也最适合便宜模型长期维护的方案。
